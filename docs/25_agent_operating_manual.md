# エージェント運用マニュアル

## 1. 目的

本書は、ボイス見積プロジェクトを複数エージェントで並列に進めるための運用ルールです。

狙いは、開発、QA、帳票、AI音声、Growth、営業/CSが互いに待たず、かつ他者の作業を壊さずに成果物を積み上げることです。

## 2. 最優先ルール

- 担当範囲外のファイルを編集しない
- 他者の変更を戻さない
- 既存ファイルを編集する前に現在内容を読む
- 仕様、実装、検証、残課題を分けて報告する
- セキュリティ、金額、帳票、AI反映に関わる変更は完了条件を厳しくする
- 分からないことは推測で断定せず、前提として明記する

## 3. 作業開始時の確認

作業を始める前に、最低限以下を確認します。

| 確認項目 | 内容 |
| --- | --- |
| 担当範囲 | 編集してよいファイル、触ってはいけないファイル |
| 目的 | 何を成果物として出すのか |
| 参照資料 | 関連するdocs、agent定義、既存コード、既存記事 |
| 依存関係 | 他チームから受け取るもの、渡すもの |
| 完了条件 | 何を満たしたら完了か |
| 禁止事項 | セキュリティ、帳票、AI、営業表現のNG |

## 4. 標準ワークフロー

1. 担当範囲と禁止範囲を確認する
2. 参照資料と既存ファイルを読む
3. 変更方針を決める
4. 担当ファイルだけを編集する
5. 可能な範囲で確認する
6. 変更ファイル、確認結果、残課題、次アクションを報告する

## 5. チーム別の運用

### 5.1 Product Core

対象:

- DB/RLS
- 認証
- API
- 見積計算
- tenant分離

完了条件:

- `tenant_id`をクライアントから信用していない
- 未ログインで保護APIにアクセスできない
- 他社ID指定を拒否する
- validationとerror responseがある
- 金額計算の根拠と端数処理が説明できる

引き継ぎ:

- UIへAPI schemaとエラー形式を渡す
- QAへRLS、権限、金額計算の確認観点を渡す
- 帳票へ見積計算結果の仕様を渡す

### 5.2 UI/UX

対象:

- 見積編集画面
- 顧客選択
- 音声入力UI
- 明細編集
- 単価マスター画面
- Excel/PDF出力導線

完了条件:

- PCとスマホの主要導線が成立する
- 保存成功、保存失敗、ローディング、空状態がある
- 長文、100明細、0円行などの表示崩れを考慮している
- AI候補はユーザー確定前に反映しない

引き継ぎ:

- APIへ必要なendpoint、payload、エラー形式を返す
- QAへ確認導線と未確認端末を渡す
- AI音声へ候補表示と確定操作の要件を渡す

### 5.3 AI音声

対象:

- 文字起こし
- 明細候補抽出
- 単価マスター照合
- 確認質問
- 評価ケース

完了条件:

- AIが顧客、見積日、担当者、単位を勝手に決めない
- 明細候補、確認質問、信頼度、候補理由を分ける
- ユーザー確定前に見積へ反映しない
- 失敗時に手入力へ戻れる

引き継ぎ:

- APIへJSON schemaを渡す
- UIへ候補表示、確認質問、信頼度表示の要件を渡す
- QAへ発話サンプルと期待結果を渡す

### 5.4 帳票/Excel

対象:

- 単価マスターExcel取込
- 外部品目ID
- 見積Excel出力
- PDF出力
- 帳票テストデータ

完了条件:

- Excel取込の列マッピング、プレビュー、エラー行が明確である
- Excel出力には業者指示事項を含める
- PDF出力には業者指示事項を含めない
- 金額と税額が一致する

引き継ぎ:

- UIへ取込プレビューとエラー表示の要件を渡す
- APIへ取込/出力schemaを渡す
- QAへ差分確認観点を渡す

### 5.5 QA/安全性

対象:

- RLS
- API権限
- AI未確認反映
- 帳票漏えい
- 金額誤り
- UI崩れ

完了条件:

- Critical/High観点を網羅する
- 再現手順、期待結果、実際結果、重大度、修正案を出す
- 修正後の再確認結果を残す

引き継ぎ:

- 各チームへ重大度つきで返す
- 統括へ未解決Critical/Highを報告する

### 5.6 Growth

対象:

- LP
- SEOブログ
- SNS
- 動画台本
- 競合比較
- 導入事例風素材

完了条件:

- ターゲット読者が明確である
- CTAと導入効果が具体的である
- 誇大表現がない
- SEO記事は`.claude/agents/seo-blog-team.md`のルールを満たす

引き継ぎ:

- UIへLP反映文言を渡す
- 営業/CSへ商談で使える比較、FAQ、導入効果を渡す
- SEO/運用へslug、meta、内部リンク候補を渡す

### 5.7 営業/CS

対象:

- 5分デモ台本
- 15分商談台本
- FAQ
- 反論対応
- 導入チェックリスト

完了条件:

- 商談でそのまま読める
- 顧客不安に具体的に答えられる
- 実装済み/未実装の境界が明確である
- 導入準備物が分かる

引き継ぎ:

- Growthへ訴求改善の材料を渡す
- Product Coreへ顧客からの要望や懸念を渡す

## 6. 横断ゲート

以下は、どのチームでも完了前に確認します。

| ゲート | 確認内容 |
| --- | --- |
| セキュリティ | 他社データが見えない、認証なしで使えない |
| AI安全性 | AI推測が未確認で保存されない |
| 帳票 | Excel/PDFの出力差分が守られている |
| 金額 | 小計、税額、合計、端数が説明できる |
| UI | PC/スマホ、長文、エラー、ローディングが考慮されている |
| Growth | 誇大表現、存在しない実績、未確定料金の断定がない |
| 報告 | 変更ファイル、確認結果、残課題、次アクションがある |

## 7. 禁止事項

- 担当外ファイルを編集する
- 他者の編集を戻す
- `git reset --hard`や強制的な巻き戻しを行う
- 仕様未確定の内容を実装済みとして営業資料に書く
- AI候補をユーザー確認なしで確定する
- PDFに業者指示事項を出す
- 他社データを参照できる可能性を残す
- 金額計算の確認なしで完了する
- テスト未実行なのに実行済みと書く
- 存在しない導入実績、保証、連携を断定する

## 9. 統括から各チームへの依頼テンプレート

```text
あなたは{チーム名}です。
担当範囲は{編集可能範囲}です。
参照資料を読み、{成果物}を作成してください。
他者の編集を戻さず、担当外ファイルは触らないでください。
完了条件は docs/25_agent_operating_manual.md と .claude/agents/mitsumori-execution-team.md に従ってください。
最後に、成果物、変更ファイル、確認結果、残課題、次アクションを報告してください。
```

SEOブログチームへ依頼する場合は、以下を追加します。

```text
記事作成時は docs/18_blog_article_ai_generation_prompt.md と .claude/agents/seo-blog-team.md を最優先ルールとして扱ってください。
画像、表、FAQ、内部リンク、CTA、禁止表現チェックを満たさない記事は完了扱いにしないでください。
```
